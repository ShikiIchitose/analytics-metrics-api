# analytics-metrics-api 開発サマリー

## 3行要約

- FastAPI / DuckDB / Parquet を用いて、KPI を返す analytics API(分析API) を実装しています。
- offline-first、CI(continuous integration: 継続的インテグレーション)、指標定義の文書化、手動検証用 SQL を通じて、再現性・検証可能性・説明可能性を重視しています。
- `url-monitor`、`exoplanet-analysis-report` に続く第3弾として、CLI ツール → 分析パイプライン → analytics API へと段階的に発展させています。

## このリポジトリで示したいこと

このリポジトリでは、単に FastAPI で API を作るだけでなく、次のような実務的な観点を重視して開発しています。

- 分析・KPI を API として提供するための設計力
- DuckDB / Parquet を使ったローカル中心のデータ処理設計
- テスト、CI、ドキュメントを含めた再現性のある開発
- 指標の意味や集計ロジックを明確に説明する documentation hygiene(文書整備の丁寧さ)
- 実装だけでなく、検証しやすさ・保守しやすさを意識した改善の積み重ね

個人ポートフォリオではありますが、実務で求められる「動くものを作る力」と「それを継続的に改善・説明する力」の両方が伝わることを意図しています。

## このリポジトリに至る流れ

このリポジトリは、これまでに公開してきた Python ポートフォリオの流れを踏まえて設計しています。  
単発で別々のものを作っているのではなく、前段の実装で得た知見を次のリポジトリに引き継ぎながら、段階的に対象領域を広げています。

### 第1弾: `url-monitor`

`url-monitor` では、Python CLI(command-line interface: コマンドラインインターフェース) として、URL の可用性や応答時間を確認し、結果を Markdown / JSON として出力するツールを実装しました。  
このリポジトリでは、HTTP request(HTTPリクエスト) の基本、CLI 設計、`pytest` によるテスト、`ruff` による静的解析、GitHub Actions CI(continuous integration: 継続的インテグレーション)、reproducible outputs(再現可能な出力) など、Python プロジェクトを継続的に保守するための基礎を固めました。

**ここで得た主な知見**

- Python で小さくても完結したツールを設計・公開する流れ
- テスト、lint、CI を含めて公開品質まで持っていく基本動線
- 出力形式を揃え、再現しやすい成果物を残すことの重要性
- 「動く」だけでなく「検証できる」状態で公開する姿勢

### 第2弾: `exoplanet-analysis-report`

`exoplanet-analysis-report` では、外部データを取得し、前処理から集計・可視化・レポート生成までを一貫して行う analytics pipeline(分析パイプライン) を実装しました。  
DuckDB と Parquet を使ったローカル中心のデータ処理、run metadata(実行メタデータ) の保存、reproducibility(再現性) の担保、preprocessing(前処理) と basic data analysis(基礎的データ分析) の流れを通じて、単なるスクリプトではなく、分析処理全体を構造化して扱う経験を積みました。

また、再現性の面では、単に同じ処理を再実行できるだけでなく、入力データや生成物の同一性を hash(ハッシュ) で確認できるようにし、実行条件・出力結果・データの対応関係を追跡しやすい形に整備しました。  
これにより、「同じコードを動かしたか」だけでなく、「同じ入力に対して同じ成果物が得られているか」を確認しやすくし、分析結果の検証可能性と traceability(追跡可能性) を高めています。

**ここで得た主な知見**

- データ取得からレポート生成までをつなぐ pipeline 設計
- DuckDB / Parquet を使った軽量で再現性の高い分析基盤
- run metadata を残すことによる検証可能性と追跡性の向上
- hash を用いて入力・出力の同一性を確認し、再現性をより厳密に扱う考え方
- 分析結果そのものだけでなく、前処理と実行条件を含めて説明する重要性

`exoplanet-analysis-report` では、DuckDB を主に分析パイプラインの実装基盤として利用し、データ取得・前処理・集計・レポート生成を再現可能な形でつなぐ経験を積みました。  
その際、run metadata や hash によって、実行条件と成果物の対応関係を追跡しやすくし、再現性と検証可能性を高める考え方も強く意識しました。

一方で `analytics-metrics-api` では、DuckDB を単なる実行基盤として使うだけでなく、SQL(Structured Query Language: 構造化問い合わせ言語) の構文や集計ロジックそのものにより深く踏み込み、KPI をどのような条件・粒度・時間範囲で数えているのかを、API 実装と手動検証用 SQL の両面から明示的に扱うようにしました。

特に本リポジトリでは、`COUNT(DISTINCT ...)`、`GROUP BY`、`WITH`、`date_trunc()` などを用いて、指標の意味を SQL レベルで追跡・検証できるようにし、「DuckDB を使った」段階から「DuckDB / SQL で指標定義そのものを設計・説明する」段階へ発展させています。

### 第3弾: `analytics-metrics-api`

本リポジトリ `analytics-metrics-api` は、上記 2 つの流れを受けて、分析・集計ロジックを API として提供する段階に進めたものです。  
`url-monitor` で培った Python プロジェクトの基礎的な保守・品質管理の流れと、`exoplanet-analysis-report` で培った DuckDB / Parquet を中心とした分析基盤・再現性重視の考え方を組み合わせ、KPI を API として扱う構成へ発展させています。

このリポジトリでは特に、次の点を意識しています。

- KPI を返す API を設計すること
- metric semantics(指標の意味) を文書として明示すること
- SQL レベルで指標ロジックを検証できるようにすること
- 実装、テスト、CI、ドキュメントを一体として継続的に改善すること

**ここで得た主な知見**

- DuckDB / Parquet を使った分析基盤を、API 実装に接続する設計
- SQL の構文と集計意味を、実装レベルで扱う経験
- `COUNT(DISTINCT ...)`、`GROUP BY`、`WITH`、`date_trunc()` などを用いた KPI 集計ロジックの設計
- API が返す値を SQL でも追跡できるようにし、説明可能性と検証可能性を高める考え方
- 実装、ドキュメント、手動検証用 SQL を組み合わせて保守しやすくする開発姿勢

つまり、  
**CLI ツール → 分析パイプライン → analytics API(分析API)**  
という流れで、扱う対象を段階的に広げながら、再現性・検証可能性・説明可能性を一貫して重視してきたことが、このリポジトリの背景にあります。

## 開発方針

このリポジトリは、以下の方針で段階的に育てています。

- **小さく公開し、継続的に改善する**  
  最初から過剰に大きくせず、MVP(Minimum Viable Product) を公開した上で、意味のある改善を小さな単位で積み上げる方針です。

- **offline-first を重視する**  
  ポートフォリオとして誰でも再現しやすいように、外部依存を最小限にし、ローカルで検証しやすい構成を重視しています。これは再現性のためだけでなく、実在の外部サービスやシステムに不要な負荷をかけないようにするためでもあります。

- **KPI の意味と SQL / API の整合を明示する**  
  「何を数えているか」が曖昧にならないように、METRICS 文書や手動検証用 SQL を整備し、指標の意味を説明可能な状態に保つことを重視しています。

- **実装・テスト・CI・文書をセットで整える**  
  コード本体だけでなく、テスト、CI、README、補助資料まで含めて、実務に近い開発の流れを意識しています。

## 主な更新履歴

### 2026/3/8 — v0.1.0 公開

**概要**  
FastAPI / DuckDB / Parquet を用いた、offline-first のプロダクト分析向けメトリクス API として v0.1.0 を公開しました。  
RESTful API として最低限のエンドポイントを持ち、KPI をローカルの Parquet データから取得できる構成を実装しています。あわせて offline-first tests と CI を整備し、最低限の再現性と検証可能性を持つ状態で公開しました。

**この更新の意図**  
まずは「分析用 API の最小実装を、ローカルで再現可能な形で公開する」ことを優先しました。  
個人開発では完成度を上げようとして公開が遅れがちですが、本リポジトリでは MVP を早期に公開し、その後の改善履歴も見える形にする方針を採っています。

**見てほしい点**

- FastAPI を使った API 設計の基礎
- DuckDB / Parquet を利用した軽量な分析系データ基盤
- offline-first tests による再現性の担保
- 個人開発でも CI まで含めて整備する開発姿勢

**示せるスキル**  
Python backend, API design, DuckDB / Parquet, testing, CI, reproducibility

---

### 2026/3/9 — v0.1.1 ドキュメントと CI の強化

**概要**  
`METRICS.md` / `METRICS.ja.md` を追加し、KPI の意味と metric semantics を明文化しました。  
あわせて `README.md` で KPI の位置づけを補足し、GitHub Actions CI に Pyrefly を追加、README / README.ja.md に CI チェック内容も追記しました。これにより、ドキュメント整備に加えて、静的型チェックを継続的に実行する体制を整え、実装の堅牢性を高めました。

**この更新の意図**  
分析系の API は「値が返る」だけでは不十分で、**その値が何を意味するか** を説明できることが重要です。  
この更新では、メトリクス定義の曖昧さを避けるために文書面を強化し、さらに CI 上で静的型チェックを導入することで、型やインターフェースの整合性を継続的に確認できるようにしました。こうした発想の背景には、研究で長く扱ってきた Fortran のように、型や入出力の整合性を重視する実装経験もあります。

**見てほしい点**

- KPI / metric semantics をドキュメントとして明示していること
- 英語版・日本語版の両方を用意し、説明責任を意識していること
- CI に静的型チェックを組み込み、堅牢性を高めていること
- 「何を作ったか」だけでなく「どう説明し、どう守るか」まで含めて改善していること

**示せるスキル**  
Technical documentation, metric design, CI and quality checks, static type checking, type and interface consistency, maintainability

---

### 2026/3/12 — 手動検証用 SQL の追加

**概要**  
`sql/debug/` を追加し、`dau`、`new_users`、`conversion_rate` のロジックを DuckDB 上で直接確認するための manual validation SQL files(手動検証用SQL) を整備しました。

**この更新の意図**  
API 実装の中に SQL が閉じているだけだと、指標ロジックを外から検証しにくい場面があります。  
そこで、アプリケーション本体とは別に、ローカルの Parquet データセットを対象とした手動検証用 SQL を用意し、集計ロジックを独立に確認しやすくしました。

**見てほしい点**

- KPI を「返す」だけでなく、「どう数えているか」を追えるようにしていること
- SQL レベルでの検証導線を用意し、デバッグ性・説明可能性を高めていること
- README にも `sql/debug/` の目的を追記し、補助資産の位置づけを明確にしていること

**示せるスキル**  
SQL validation, analytics logic inspection, DuckDB / Parquet workflow, maintainability, documentation hygiene
